# data/ — Runtime Data Storage

This folder holds runtime data generated by The Hunter during pipeline execution.
It does **not** contain the training datasets — those are downloaded automatically
from Kaggle at notebook runtime via `kagglehub`.

---

## Directory Structure

```
data/
└── threat_memory.db        # SQLite persistent threat memory database (auto-created)
```

---

## File Details

### threat_memory.db

| Property | Value |
|----------|-------|
| **Type** | SQLite 3 database |
| **Size** | ~12 KB (grows with use) |
| **Created by** | `hunter_gradio_app.py` and the training notebook on first pipeline run |
| **Committed to GitHub** | Yes — contains demo run data only, no sensitive information |

#### Purpose

`threat_memory.db` is The Hunter's persistent institutional memory. Every time Agent 3
(the SOC Orchestrator) issues a verdict, the result is written to this database. On
subsequent emails from the same sender, Agent 3 queries this history before deciding
on a response — enabling the **repeat-offender escalation** behavior.

#### Schema

The database contains a single table:

```sql
CREATE TABLE threat_log (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    sender      TEXT    NOT NULL,
    risk_score  REAL    NOT NULL,
    action      TEXT    NOT NULL,
    timestamp   TEXT    NOT NULL
);
```

| Column | Description |
|--------|-------------|
| `id` | Auto-incrementing primary key |
| `sender` | Sender identifier extracted from the email (text field or "unknown") |
| `risk_score` | Ensemble risk score at time of verdict (0.0 – 1.0) |
| `action` | Verdict: `ALLOW`, `ALLOW AND LOG`, `FLAG WITH WARNING`, or `QUARANTINE` |
| `timestamp` | UTC ISO 8601 timestamp of when the verdict was logged |

#### Repeat-Offender Logic

When Agent 3 processes an email, the `ThreatMemoryTool` queries the last 5 entries
for the sender. If any prior verdict was `FLAG WITH WARNING` or `QUARANTINE`, the
sender is flagged as a **repeat offender** and the defense policy automatically
escalates one tier:

- A sender who would receive `FLAG WITH WARNING` is escalated to `QUARANTINE`
- A sender who would receive `ALLOW AND LOG` is escalated to `FLAG WITH WARNING`

#### Location Configuration

The database path is configured in `hunter_gradio_app.py`:

```python
DATA_DIR  = Path("data/")
THREAT_DB = DATA_DIR / "threat_memory.db"
```

And in the training notebook under **Section: Configuration**:

```python
DB_PATH = "data/threat_memory.db"
```

Both the app and the notebook write to the same `data/threat_memory.db` file,
so threat history accumulates across both the web dashboard and notebook demo runs.

#### Resetting the Database

To clear all threat history and start fresh, either delete `threat_memory.db` entirely
(it will be recreated on next run) or run:

```python
import sqlite3
with sqlite3.connect("data/threat_memory.db") as conn:
    conn.execute("DELETE FROM threat_log")
```

## Training Datasets (Not Stored Here)

The training datasets are **not included** in this folder because of their size.
They are downloaded automatically at runtime.

| Dataset | Source | How to Access |
|---------|--------|--------------|
| **Alam Dataset** | Kaggle — `uciml/sms-spam-collection-dataset` variant | Downloaded automatically via `kagglehub` in notebook **Section: Data Acquisition - Alam Dataset** |
| **Cratchley Dataset** | Kaggle — secondary phishing dataset | Downloaded in notebook **Section: Secondary Dataset Analysis - Cratchley** |

To use `kagglehub`, a Kaggle API key must be configured. See **Section: Secure Credential Loading**
in the notebook, or add `KAGGLE_KEY` / `KAGGLE_USERNAME` to your `.env` file.

If `kagglehub` is not available or the key is missing, the notebook will prompt for a manual
upload of the CSV files. Refer to `QUICKSTART.md` for the manual upload path.
